01 About

수강생이 어디서 막히는지,
현장에서 반복해서 확인해 온 사람

방산 엔지니어 → 대면 전달 → AI 프로덕트 빌더 — 세 번의 전환 동안 항상 같은 일을 해왔습니다. 같은 내용을, 상대의 배경·이해도·심리에 따라 설명의 순서와 호흡을 다시 짜는 일. 수강생이 도구보다 먼저 문제 정의에서 막힌다는 감각은 수백 명을 대면으로 마주 보며 반복해서 얻은 것입니다.

2014.03 — 2017.06 3년 4개월

정확성과 검증이 체화된 시기

한화오션(주) · 특수선무장설계팀 · 통신/무장 기본설계

장보고-III 잠수함 프로젝트에서 독일 제조사(영어 핫라인)와 조달·배관·전장설계부서 사이의 생산설계 이슈를 조율하고, 국방과학연구소 보고용 자료를 반복해서 취합·정리했습니다. 외부 기관에 올라가는 자료는 수치 하나, 용어 하나 틀리면 끝이라는 긴장 아래에서 일했습니다.

이때 체화된 정확성·검증 중심의 태도는 지금 LLM 파이프라인을 설계할 때, 그리고 강의에서 설명이 흔들리지 않게 고정할 때 그대로 쓰이는 근력이 됐습니다. 한서대학교 항공운항학과 5개 학기(학점 4.36 / 전체 석차 1위)로 전공 전환을 완수한 학습 패턴과 이어집니다.

2022.04 — Present 4년+

수백 명의 막힘 지점을 읽어낸 4년

대치동 정시 컨설팅 · 프리랜서 컨설턴트

매년 바뀌는 입시제도 안에서 매 시즌 수백 명의 학생·학부모를 대면 컨설팅했습니다. 사전 설문 → 대면 상담 → 후속 관리 → 파이널 콜로 이어지는 고밀도 프로세스를 매 시즌 안정적으로 운영했고, 결과적으로 학원 재등록·지인 추천의 리텐션으로 이어졌습니다.

이 환경에서 가장 깊이 배운 것은 수강생은 도구보다 먼저 문제 정의에서 막힌다는 감각입니다. 같은 내용도 상대의 배경·이해도·심리에 따라 설명의 순서와 호흡을 매번 다시 짜야 한다는 것을 수백 번 반복해서 확인했습니다. 대학 시절에는 6개월간 고등학생 대상 학원 집합 강의를 진행하며 1:1 상담과는 다른 집합 포맷의 호흡을 별도로 검증했습니다.

코드잇 공고의 "수강생이 어떤 것을 어려워하는지 잘 느끼고 공감"에 직접 대응하는 4년의 근력입니다.

에듀테크 스타트업 재직 시절 숏폼 트렌드 크롤링·분석 데스크톱 앱과 멀티 AI 영상 제작 데스크톱 앱을 0→1로 단독 구축해 배포했습니다. 이후 프리랜서로 전환해 오프라인 이벤트용 실시간 매칭 웹앱을 설계·배포하고 지금도 유지보수 중입니다.

이 과정에서 어느 모델이 어떤 작업에 강한지, 파이프라인 어느 단계에서 품질이 무너지기 쉬운지, API 비용과 출력 품질의 트레이드오프를 어떻게 설계하는지 같은 문제를 실제로 다뤘습니다. 수강생에게 단순한 기능 소개가 아니라 실제 업무에 적용할 때 어디에서 막히는지를 현장 감각으로 전달할 수 있습니다.

가르칠 자격은 "배운 사람"이 아니라 "직접 만들어 본 사람"에게서 생깁니다. AX · AI 활용 · AI 기초 세 레벨 모두에 바로 연결되는 경험을 갖고 있습니다.

02 Teaching Topics

빠르게 습득해 가르칠 수 있는 주제

코드잇 지원 서류 [2]번 문항에 대한 답변을 포트폴리오 위에 그대로 펼쳐놓았습니다. 모두 실제로 만들어 운영한 프로덕트와 직접 실험해 본 설계 판단에 기반합니다. AX · AI 활용 · AI 기초 세 레벨에 직접 대응합니다.

AI 기초 · Module 01

생성형 AI 이해와 실무 입문

Target AI를 처음 배우는 비전공자 · 일반 실무자 · 입문 수강생

생성형 AI가 무엇이고, 어디까지 할 수 있으며, 어디서 한계가 생기는지를 헷갈리지 않게 쪼개서 설명합니다. 프롬프트 몇 개를 외우는 방식이 아니라, "어떤 문제에 어떤 도구를 왜 쓰는가"를 기초부터 잡아 주는 과정. 대치동 4년간 비전문가에게 복잡한 제도를 풀어 설명해 온 전달 감각을 그대로 적용합니다.

AI 활용 · Module 02

업무 생산성 · 자동화 실습

Target 기획·마케팅·운영·디자인 등 비개발 실무자

문서 정리·리서치·초안 작성·반복 업무 자동화까지 — 내일부터 바로 적용 가능한 활용법을 실습 중심으로 다룹니다. 참가자 자신의 실제 업무를 예제로 사용해, "강의실에서 배운 것"이 아니라 "업무에서 써먹는 것"이 남도록 설계합니다. Claude · ChatGPT · Gemini 역할 분담 원칙도 함께.

AX · Module 03

조직 도입과 사람·도구 역할 설계

Target 팀 리드 · 중간관리자 · HRD · 사내 AX 담당자

AX는 도구 소개가 아니라 업무 재설계의 문제라는 전제에서 출발합니다. "무엇을 AI에게 맡기고, 무엇을 사람이 판단해야 하는가"를 실제 프로덕트에 의도적으로 남겨 둔 Human-in-the-loop 사례를 기반으로 설명합니다. 영상 자동화 앱에 별도 팩트 체크(Perplexity) 단계를 넣은 판단도 여기에 포함됩니다.

  • 생성 AI 도입 타당성 판단 프레임워크AX

왜 최신 Realtime 대신 2초 폴링을 1차 채널로 두었나?

현장 Wi-Fi 품질이 일정하지 않고 WebSocket이 수 초 끊기면 라운드 오픈·투표 마감 같은 타이밍 크리티컬한 순간에 UI가 stale해집니다. 일부 시나리오에서 Realtime 안정성이 행사 운영 요구치에 못 미쳐, 핵심 화면은 2초 폴링을 1차 채널로, Realtime을 보조로 두는 hybrid 구조로 전환했습니다. 재접속 시 SHA-256 해시 기반 resume_token으로 세션 복구, 동시 RPC에는 ON CONFLICT 멱등 처리 + advisory lock으로 race condition을 방어합니다.

React 19 TypeScript Supabase PostgreSQL Zustand 5 Vite 6 Realtime + Polling RLS bcrypt Admin
교육 포인트 이 사례로 실시간 웹앱 설계 · 동시성 제어 · 운영 환경 트레이드오프를 가르칠 수 있습니다. "Realtime을 무조건 쓰면 안 되는 이유"와 "장애 복구 구조를 언제부터 설계에 넣어야 하는가"는 실제 운영 경험 없이는 전달이 어렵습니다.

Design Decision
왜 완전 자동화가 아니라 Human-in-the-loop 구조로 남겼나?

전 공정 자동 생성은 구현이 쉽지만, YouTube 등 플랫폼이 완전 자동 생성 콘텐츠 제재를 점점 강화하는 방향으로 움직이고 있었습니다. 짧은 편집 개입 구간을 의도적으로 남겨, 스크립트 수정·팩트 보정·최종 승인을 사람이 통과시키는 HITL 게이트를 유지했습니다. 자동화 효율을 일부 포기하는 대신 리스크를 플랫폼이 아니라 사람과 나눠 가지는 설계입니다.

Design Decision
왜 Electron 앱에서 API를 직접 호출하지 않았나?

데스크톱 앱은 코드가 클라이언트에 통째로 배포됩니다. API 키를 번들에 심으면 asar 언패킹·메모리 덤프만으로 평문 키가 그대로 노출됩니다. 클라이언트에는 사용자 JWT만 두고 실제 API 호출은 Supabase Edge Functions 프록시가 대신 수행 — 전형적인 BFF(Backend for Frontend) 패턴입니다. 추가 레이턴시 수십 ms를 감수하고 키 노출 방지 + 서버 측 레이트 리밋·쿼터·로그 통합 제어를 얻었습니다.
